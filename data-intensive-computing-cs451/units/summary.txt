Physical View

User submits a job to the cluster
The cluster schedules the map and reduce tasks - the number of map tasks is derived from the size of the input file(s). The number of reduce tasks is set by the user
The workers start reading and mapping their split of the file
As they map, they save the “intermediate” key-value pairs to the local file system, not to the network file system
When the job is nearly finished, reducer workers start asking mappers for intermediate files that are ready
Once a reducer has all of its intermediate files it begins applying reduce - the output is written to the network file system
Distributed Group By in MapReduce

Map: Map outputs are buffered in memory in a circular buffer. When buffer reaches threshold, contents are spilled to disk. Spillers are merged into a single, partitioned file (sorted within each partition). Combine runs (or might run) during the merges.

Reduce: First, map outputs are copied over to reducer machine. Sort is a multi-pass merge of map outputs (happens in memory and on disk). Combiner runs during the merges. Final merge pass goes directly into reducer.

The Combiner in the Reducer usually runs when waiting for some mappers to finish (when some are already done). Not as advantageous as running combiner in Mapper though.

Question

How do we get data to the workers

Compute nodes communicate with SAN: Storage Area Network

This makes sense for compute-intensive tasks as computations are likely to take long while even on sophisticated hardware. Communication costs are outweighed by computation costs. For data-intensive tasks, computations aren’t likely to take nearly as long, so computation costs are outweighed by the communication costs. Likely to experience latency even with high speed transfer.

Solution is to not move data to workers, but move workers to data.

Key idea: Co-locate storage and compute. Start up worker on nodes that hold the data.

If a server is responsible for both data storage and processing, Hadoop can do some optimization.

On a Hadoop cluster, resource manager is in charge of running tasks. The resource manager assigns jobs to workers that already have relevant data (so it talks to the NameNode).

Under the hood: DAG scheduler

General task graphs that automatically pipeline functions. Data locality aware and partitioning aware to avoid shuffles. A job is broken down to multiple stages that form a DAG.

You can get the DAG from an RDD using the toDebugString method. It’s also viewable through the Hadoop monitoring page.

Narrow dependencies are faster than wide dependencies because it does not require shuffling data between working nodes.

reduceByKey, groupByKey will have narrow dependencies if the upstream RDD is already partitioned by key.

Since Spark avoids heavy disk i/o, it significantly improves the performance (relative to Hadoop).

Spark outperforms Hadoop in iterative programs because it tries to keep the data that will be used again in the next iteration in memory. In contrast, Hadoop always read and write from/to disk.

Random walk interpretation

Imagine a random web surfer. At any time t, surfer is on some page i. At time t+1, the surfer follows an out-link from i uniformly at random. Ends up on some page j linked from i. Process repeats indefinitely.

There is not actually a random walker. We construct a distribution of what page the random walker may be on.

Let p(t)… vector whose ith coordinate is the probability that the surfer is at page i at time t. So p(t) is a probability distribution over pages.

Lecture 11
PageRank Problems:

Some pages are dead ends (have no out-links). Random walk has “nowhere” to go to. Such pages cause importance to leak out. Not connected ⟹ not ergodic.

Spider traps (all out-links are within the group). Random walker gets stuck in a trap, and eventually the spider trap absorb all importance. Not connected ⟹ not ergodic.

Solution: Teleports

The Google solution for spider traps: At each time step, the random surfer has two options.

With probability β, follow a link at random
With probability (1−β), jump to some random page
Common values for β are in the range 0.8 to 0.9
Surfer will teleport out of spider trap within a few time steps.

Solution: Always Teleport

Teleports: Follow random teleport links with probability 1.0 from dead-ends. Adjust matrix accordingly.

Question

Why Teleports solve the problem?

Why are dead-ends and spider traps a problem, and why do teleports solve the problem?

Spider-traps are not a problem, but with traps, PageRank scores are not what we want. Solution: Never get stuck in a spider trap by teleporting out of it in a finite number of steps.

Dead-ends are a problem. The matrix is not column stochastic, so our initial assumptions are not met. Solution: Make matrix column stochastic by always teleporting when there is nowhere else to go.

k-means clustering.

We assume Euclidean space, start by picking k, the number of clusters, and initialize clusters by picking one point per cluster.

For each point, place it in the cluster whose current centroid it is nearest.

After all points are assigned, update the locations of centroids of the k clusters.

Reassign all points to their closest centroid. Sometimes moves points between clusters.

We repeat the last two steps until we converge. Convergence happens when points don’t move between clusters and centroids stabilize.

Relational Joins

Inner Join (⋈)

One-to-one joins: No crossing is needed, memory requirements are minimal. Modified merge sort.

One-to-many joins: For the one side, hold the tuple in memory, cross with all tuples from the many side with matching join key. One pass, minimal memory usage.

Many-to-many joins: Need to hold all tuples with a given key in memory for both sets. Cross one set agains the other. Bad when both sets are large, and the many is large.

For Inner Join on MapReduce there are 3 options. They are ordered by goodness.

Hash Join
Map-Side Join
Reduce-Side Join
Hash Join: If one set is so small i t easily fits into memory of a single node then, load it into every mapper as a hash table, each mapper joins its split against the hash table, map-side only, can be pipelined. This is fast

Map-Side Join: If both files are sorted by key and split by the same partitioner, in that case mappers can join row by row.

Reduce-Side Join: Mappers - MultipleInputFiles class. Basic idea is that each mapper emits its stuff, key = join key. Reducer gets keys in sorted order, easy to join them together.

Reduce-Side 1-to-1 join: Reducer always gets 1 key per subquery. Join is straightforward. Need to tag each tuple with its source, but the sort order doesn’t matter.

Reduce-Side 1-to-many join: Reducer holds the tuple from the 1-side in memory, then joins with all of the many-side tuples with the same key.

Sliding vs tumbling windows.

Sliding window example can be the last minute of data, updated every 5 seconds. Each window has most data in common with previous.

Tumbling window could be last minute of data, updated every minute. We have a blank slate, no data in common between windows. We might be able to make a sliding window out of smaller tumbling windows.

Windows based on counts have a window of n records over the stream. When the stream is slow, windows are long. When the stream is fast, windows are very short.

Challenges of stream approach:

Inherent challenges are latency requirements and memory/storage. Framework challenges are bursty nature of streams, load balancing and clustering, out-of-order delivery.

Two hard problems with distributed delivery are exactly-once deliver and out-of-order delivery.

We have producers/consumer models. Producer generates/gathers/aggregates data, consumer process it.

Producer can send to a consumer (push/callback method), or the consumer can request data from the producer (pull/poll method).

A consumer can consume data from multiple producers, and a producer might need to send data to multiple consumers. How do we scale this?

We need middleware. A data broker sits between producers and consumers and acts as a pub/sub queue. Producers push to the broker, broker pushes to consumers. We could also have a push pull method where the producer pushes to the broker, the consumer pulls from the broker as needed. An advantage of consumer polling is that the broker doesn’t need to worry about spotty connections by consumers. Disadvantage is that frequent polling for infrequently arriving data is wasteful.

Kafka is a general purpose data broker. Created by LinkedIn, named after the author Franz Kafka.

Kafka has up to 2 million writes per second on 3 cheap machines. Kafka is fast. Even though all data is persisted to disk, all writes go to the page cache of OS. Very fast reads since it’s efficient to transfer data from page cache to a network socket.

Kafka stores data by a topic. Kafka topics are split into partitions, and are replicated across the cluster. Brokers are leaders for partitions, and other brokers are follower for that same topic. Kafka can tolerate R−1 failures without data loss where R is the amount of replication.

Messages from the producer are hashed, or balanced round-robin and appended to the partitions. Operations are idempotent. Producers send a unique key with each batch of message. Brokers log these and reject duplicate messages.

Partitions are totally ordered, but topics are not. If events have a key then all events with the same key are in the same partition, and therefore are in order by arrival time. If events have no key, they are not in exact order anymore.

A partition can be read by multiple consumers. Each consumer remembers its offset in each partition. Consumers are responsible for their own bookkeeping.

Reservoir Sampling

Problem: Select S values uniformly at random from a stream of N values.

Assumption: N is very big, and not known ahead of time.

The solution is to store the first S values. When receiving kth, keep it with probability S/k. If keeping it, randomly discard one of the existing S elements to make room.

We can prove this by hand-wavy induction.

Claim: For K>=S, all values V

Basis for induction K=S+1.

Assume that for some K>S, all K values have a S/K chance of being in the reservoir.

S/(K+1)

1−(S/(K+1)×(1/S))=K/(K+1)

S/K×K/(K+1)=S/(K+1)

So for all K>S, after K values have been seen, each value has a uniform S/K chance of being retained in the reservoir.

Bloom Filters

Problem: Set is too large to hold, want to know if it contains X.

Solution: A bit vector of m bits, and k unique hash functions.

Put: We take our input, put it through the hash functions where our output is between [0,m]. Set the resulting bits in the vector to 1.

Contains: Take the input, run it through the hash function, and check if all of the resulting bits are set to 1.

It is obvious that this has no false negatives. If it’s been seen, all of it’s bits are set to 1. However there are false positive. The more unique elements seen, the more bits are set to 1.

Can tune false positive rate by adjusting values for m and k.

Count-Min Sketch

Counting the frequency of a value X within a multiset. Like Bloom Filters, we have a vector-length m and k independent hashes. Unlike Bloom Filters, each hash has its own int-vector of m values. We increment the bits specified by the output from the hash functions by 1 in the matrix of vectors.

When we do get, we run the input through the hash function, check all the bits and take the minimum count to be our frequency. We take the minimum since other integers may “collide” on some bits and artificially increase the count.

What if there are no partitions?

Then it is a tradeoff between consistency and latency. Caused by the possibility of failure in distributed systems. High availability → replicate data → consistency problem.

Availability and latency are arguable the same thing. Being unavailable is extremely high latency.

PACELC is a more complete description of the space of potential tradeoffs of a distributed system.

If there is a partition (P), how does the system trade off availability and consistency (A and C); else (E), when the system is running normally in the absence of partitions, how does the system trade off latency (L) and consistency (C).

PA/EL Systems: Give up both C’s for availability and lower latency. (Dynamo, Cassandra)

PC/EC Systems: Refuse to give up consistency and pay the cost of availability and latency. (BigTable, HBase)

PA/EC Systems: Give up consistency when a partition happens and keep consistency in normal operations. (MongoDB)

PC/EL Systems: Keep consistency if a partition occurs but gives up consistency for latency in normal operations. Strange. (Azure Cosmos).
